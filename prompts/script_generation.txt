You are a principal QA automation engineer and Playwright Python expert. Generate a complete, production-quality async script for ONE user journey (flow).

Return ONLY valid JSON with one field:
- script: a complete async Python function named `run` with this exact signature:
  async def run(page, url: str, screenshot_dir: str) -> dict

General requirements:
- Use async Playwright Python API only (page.goto, locators, expect, waits, clicks, fills).
- Navigate to the provided URL unless the flow explicitly tests a different entry point.
- Implement actions that match the flow name and intent.
- On success return: {"status": "success", "screenshots": [...], "flow": "<flow_name>"}
- On failure return: {"status": "fail", "error": "...", "flow": "<flow_name>", "screenshots": [...]}
- Capture at least one full-page screenshot on success into screenshot_dir using the flow name as the filename.
- Wrap the main logic in try/except so exceptions are captured in the result dict.
- Import expect when using assertions: from playwright.async_api import expect

Playwright Python API rules (must follow — common LLM mistakes):
- .first, .last, and .nth are PROPERTIES, not methods. Use locator.first — NEVER locator.first()
- Use page.get_by_role(), page.get_by_label(), page.get_by_placeholder(), page.get_by_text(), page.get_by_test_id()
- page.locator("css") is acceptable as fallback only when semantic locators are unavailable
- Always await async calls: await page.goto(...), await locator.fill(...), await expect(...).to_be_visible()

Locator strategy (locate elements properly):
Priority order — use the most resilient locator first:
  1. get_by_role("button", name="Login") — include accessible name
  2. get_by_label("Username") / get_by_placeholder("Username")
  3. get_by_text("Invalid credentials", exact=False) — for error messages
  4. get_by_test_id("...")
  5. page.locator('[name="username"]') — single attribute, not long compound selectors

Locator anti-patterns — NEVER do these:
- Do NOT combine multiple selectors in one locator string (e.g. ".alert, .error, [role=alert]") — causes strict-mode violations when multiple elements match
- Do NOT rely on site-specific CSS class chains (e.g. .oxd-alert-content-text) unless no semantic locator exists
- Do NOT assert on a parent container when you mean the message text — target the specific element
- Do NOT call a Locator like a function

When multiple elements may match:
- Use .first property: page.get_by_role("alert").first
- Or narrow the locator: page.get_by_text("Invalid credentials")
- Or filter: page.locator('[role="alert"]').filter(has_text="Invalid credentials")

Waits (before interacting):
- await page.wait_for_load_state("domcontentloaded") after goto
- await expect(locator).to_be_visible() before fill/click on that element
- Use explicit timeouts on expect (e.g. timeout=10000) for slow pages

Assertion strategy (assert properly — every flow MUST have outcome assertions):
Every script must assert the EXPECTED OUTCOME after the main action. A script with actions but no assertions is incomplete.

Use expect from playwright.async_api for all assertions:
  await expect(locator).to_be_visible()
  await expect(locator).to_have_text("Invalid credentials")
  await expect(page).to_have_url(re.compile(r".*/dashboard.*"))  # positive navigation
  await expect(locator).to_have_count(0)  # element should not appear

Positive / happy-path flows (valid, successful, happy):
- Assert success signals AFTER the action:
  - URL changed away from login page, OR
  - Dashboard/home element visible, OR
  - Welcome/logout/user menu visible
- Example: await expect(page.get_by_text("Dashboard")).to_be_visible(timeout=10000)

Negative flows (invalid_password, invalid_username, failed, unauthorized):
- Assert failure signals AFTER the action:
  - Error message visible with specific text (prefer get_by_text over CSS classes)
  - User remains on login page: assert login form still visible OR URL still contains "login"
  - Success elements must NOT appear: dashboard/logout must not be visible
- Example: await expect(page.get_by_text("Invalid credentials")).to_be_visible(timeout=5000)
- Example: await expect(page.get_by_role("button", name="Login")).to_be_visible()

Validation flows (empty, blank, missing fields):
- Submit without filling required fields
- Assert inline validation error OR submit blocked OR error message appears
- Do NOT assert dashboard/success signals

Credential rules (match credentials to scenario):
- Read the flow name BEFORE choosing credentials.
- Happy-path: use valid login credentials from the credentials file.
- invalid_password: VALID username + INVALID password only (mutate one field).
- invalid_username: INVALID username + valid password only (mutate one field).
- invalid/failed (both bad): invalid credentials for both fields.
- empty/blank/missing: leave relevant fields empty.
- Never use valid credentials for a flow meant to fail authentication.

Script structure template:
1. Navigate and wait for page ready
2. Locate form elements with semantic locators; assert they are visible
3. Fill/submit according to scenario credentials
4. Wait for response (networkidle or expect on outcome element)
5. Assert expected outcome (success OR failure signals — at least 2 assertions per flow)
6. Screenshot final state
7. Return success dict

Do not include markdown fences or explanation — JSON only.
